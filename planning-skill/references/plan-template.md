# 구현 계획서 템플릿

최종 문서는 프로젝트 루트 `docs/<date>-<topic>-plan.md`에 저장한다. 날짜는 로컬 날짜를 쓰고, `topic`은 요구사항 핵심을 짧은 slug로 정한다. 같은 요구사항의 기존 문서가 있으면 갱신 여부를 먼저 판단하고, 새 파일이 필요하면 slug를 더 구체적으로 잡아 덮어쓰기를 피한다.

```md
# <작업 제목>

## 1. 문제 맥락과 개발 의도
- Primary user: <직접 사용자, 운영자, 개발자 중 이번 변경의 핵심 수혜자>
- Pain point: <현재 무엇이 불편하거나 실패하는가>
- Why now: <왜 지금 이 변경이 필요한가>
- Intent: <이번 변경으로 달성하려는 개발 의도 또는 구현 목적>
- Expected outcome: <구현 후 사용자가 무엇을 더 쉽게, 빠르게, 안전하게 할 수 있어야 하는가>
- Constraints or assumptions: <확인이 필요한 가정이나 제약은 무엇인가>

## 2. 사용자 경험 목표와 성공 기준
- Core scenario: <가장 중요한 사용자 시나리오 또는 운영 흐름>
- Before -> After: <변경 전/후 경험 차이>
- UX guardrails: <절대 해치면 안 되는 기존 경험, 계약, 성능, 접근성, 피드백 품질>
- Success signals: <완료 시 관찰 가능한 결과 또는 측정 가능한 신호>

## 3. 확인한 공식 문서와 기술 판단
- Source: <공식 문서 또는 1차 자료>
- Relevance: <이번 요구사항과 어떤 점이 직접 연결되는가>
- Decision: <계획에 반영한 기술 결정 또는 제약>

## 4. 현재 프로젝트 작업 범위
- In scope: <영향받는 디렉터리, 파일, 모듈, 설정, 테스트>
- Existing behavior: <현재 구현에서 유지해야 할 점>
- Fixed constraints: <변경하면 안 되는 계약, 외부 규칙, 평가 기준, 플랫폼 제약>
- Dependencies: <연관 컴포넌트, 데이터, 외부 서비스>
- Unknowns: <아직 확인이 필요한 항목과 다음 확인 액션>
- Blockers: <계획 확정을 막는 미해결 항목과 해소 조건. 없으면 없음이라고 적기>

## 5. 구현 단계
1. <첫 번째 단계>
2. <두 번째 단계>
3. <필요한 경우만 추가 단계>

각 단계는 다음을 만족한다.
- 결과물이 분명하다.
- 어떤 요구사항, 성공 기준, 사용자 경험 목표를 닫는지 보인다.
- 어느 코드 영역을 건드리는지 드러난다.
- 검증 방법이 있다.
- keep/discard 판단 기준 또는 완료 신호가 있다.

단계 수는 고정하지 않는다. 작은 작업이면 줄이고, 큰 작업이면 필요한 만큼 더 쓴다.

## 6. 검증 계획
- 자동 검증: <단위 테스트, 통합 테스트, 빌드 확인 등 필요한 항목>
- 사용자 시나리오 확인: <핵심 흐름의 수동 확인 또는 시뮬레이션 절차>
- 상태별 확인: <empty/loading/error/권한 없음/오프라인 등 해당 상태 검증>
- 경험 품질 확인: <접근성, 문구/피드백, 응답성, 포커스/내비게이션 등 해당 항목>
- 각 검증 항목이 어떤 단계, 성공 기준, 리스크와 연결되는지
- 회귀 위험이 큰 부분
- 배포 전 확인 사항

## 7. 리스크와 대응
- 주요 리스크
- 대응 또는 완화 방안

## 8. Out of Scope
- 이번 요구사항에서 제외할 작업
```

문서는 설계 제안서가 아니라 실행 계획이어야 한다. 배경 설명보다 실제 변경 순서와 검증 방법을 우선하되, 문제 맥락과 개발 의도 없이 기술 변경만 나열하지 않는다. 직접적인 end-user 기능이 없는 작업이면 운영자 또는 개발자 경험을 사용자 관점으로 적는다. 각 섹션은 요구사항, 공식 문서, 실제 프로젝트 구조 중 최소 하나의 근거와 연결되어야 한다. `적절히`, `필요시`, `관련 코드`, `후속 검토`처럼 실행 기준 없는 표현은 그대로 두지 않는다. blocker가 남아 있으면 최종본으로 닫지 말고, 필요한 확인 액션을 함께 남긴다.
