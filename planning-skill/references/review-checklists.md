# 리뷰 체크리스트

## 계획 수렴 하네스

계획 작업을 아래 하네스 안에서 반복한다.

- 고정 하네스: 요구사항, 공식 문서/1차 자료, 실제 프로젝트 구조, `plan-template.md`, 이 체크리스트
- 변경 가능한 산출물: 현재 계획 작업본 1개
- Keep: blocker 또는 finding 감소, 단계/검증 명확화, 같은 범위에서의 단순화
- Discard: scope 증가만 있고 문제를 줄이지 못하는 수정, 근거 없는 추측 추가, 검증 가능성 약화
- Stop: 적절성 검토 0 finding, 과잉 설계 검토 0 finding, 독립 최종 재검토 0 finding, blocker 0

## 공식 문서 확인

다음 질문에 모두 답할 수 있을 때까지 필요한 공식 문서를 확인한다.

- 이번 요구사항에 새 기술, API, 프레임워크 제약이 있는가
- 최근 변경 가능성이 있는 정보인가
- 프로젝트가 이미 쓰는 기술의 공식 가이드 또는 1차 문서가 있는가
- 계획에 넣을 제약, 권장 패턴, 마이그레이션 포인트가 문서로 확인되었는가

공식 문서 확인 결과는 계획서의 "확인한 공식 문서와 기술 판단" 섹션으로 정리한다.

## 적절성 검토

초안 계획을 읽고 아래 finding이 있는지 점검한다.

- 요구사항의 목표 또는 성공 기준이 빠져 있다.
- 각 구현 단계가 어떤 요구사항 또는 성공 기준을 닫는지 보이지 않는다.
- 프로젝트 범위가 실제 파일, 모듈, 설정과 연결되지 않는다.
- 필요한 기술 검토가 빠졌거나 공식 문서 근거가 없다.
- 고정 하네스와 변경 가능한 산출물이 구분되지 않는다.
- 구현 단계가 모호해서 누가 봐도 바로 실행하기 어렵다.
- 구현 단계에 영향을 받는 코드 영역 또는 검증 방법이 빠져 있다.
- `적절히`, `필요시`, `관련 코드`, `후속 검토`처럼 실행 기준 없는 표현이 남아 있다.
- 검증 계획이 없거나 회귀 위험을 다루지 않는다.
- 리스크나 미확인 사항이 빠져 있다.
- `Unknowns`가 blocker 여부와 다음 확인 액션까지 구분되지 않는다.

finding이 하나라도 있으면 다음 순서로 수정한다.

1. finding을 짧고 구체적으로 적는다.
2. 계획서의 해당 섹션을 수정한다.
3. 같은 체크리스트를 다시 돌린다.

추가 finding이 없어질 때까지 반복한다.

## 과잉 설계 검토

적절성 검토가 끝난 뒤, 계획이 요구사항 대비 과하지 않은지 점검한다.

- 각 단계가 현재 요구사항과 직접 연결되는가
- "있으면 좋음" 수준의 리팩터링이 기본 계획에 섞여 있지 않은가
- 불필요한 추상화, 플랫폼 확장, 범용화가 들어가 있지 않은가
- 병렬 작업 스트림이 정말 필요한가
- 테스트/검증 범위가 요구사항 대비 과도하게 넓지 않은가
- 미래 가능성 대응이 현재 요구사항보다 앞서 있지 않은가
- finding을 줄이지 못하는 설명 또는 단계가 단지 문서 길이만 늘리고 있지 않은가

finding이 있으면 다음 순서로 수정한다.

1. 왜 과한지 한 문장으로 적는다.
2. 해당 항목을 축소, 삭제, out-of-scope 이동 중 하나로 처리한다.
3. 체크리스트를 다시 돌린다.

추가 finding이 없어질 때까지 반복한다.

## 패스 재시작 규칙

아래 항목 중 하나라도 바뀌면 이전 clean 판정은 무효다. 적절성 검토부터 다시 시작한다.

- 문서 구조나 섹션 의미
- 구현 단계 수, 순서, 단계 경계
- 고정 제약, 의존성, `Unknowns`, `Blockers`
- 검증 계획, 리스크 대응, out-of-scope 선언

## 독립 최종 재검토

적절성 검토와 과잉 설계 검토가 clean이어도 바로 종료하지 않는다. 이 패스는 앞선 finding 메모를 그대로 재사용하는 검토가 아니라, 요구사항과 계획을 처음 다시 맞춰 보는 최종 검토다.

1. 요구사항, 공식 문서/1차 자료, 실제 프로젝트 구조를 짧게 다시 확인한다.
2. 기존 finding 메모를 잠시 제쳐두고 계획서를 처음 읽는 것처럼 위에서 아래로 다시 읽는다.
3. 각 섹션마다 아래 질문으로 새 finding이 있는지 점검한다.
   - 이 항목이 왜 필요한지 요구사항, 공식 문서, 실제 코드 범위 중 어디에 근거하는가
   - 이 항목이 빠지면 구현 단계에서 어떤 혼선이나 회귀 위험이 생기는가
   - 같은 의미를 더 작고 명확하게 적을 수 있는가
4. 새 finding이 하나라도 나오면 기록 후 문서를 수정하고 적절성 검토부터 다시 시작한다.
5. 수정 없이 전체 문서가 통과할 때만 종료 조건 후보로 인정한다.

## 종료 조건

아래 조건을 모두 만족하면 계획을 확정한다.

- 적절성 검토에서 추가 finding이 없다.
- 과잉 설계 검토에서 추가 finding이 없다.
- 독립 최종 재검토에서 추가 finding이 없다.
- blocker unknown이 남아 있지 않다.
- 남아 있는 unknown은 blocker가 아니고 다음 확인 액션이 적혀 있다.
- 계획 단계마다 작업 범위와 검증 방법이 보인다.
- 각 구현 단계가 닫는 요구사항 또는 성공 기준이 보인다.
- 최종 문서를 프로젝트 루트 `docs/`에 기록할 준비가 끝났다.
- keep된 수정들만 남았고 discard해야 할 우회안이나 추측성 단계가 제거되었다.
