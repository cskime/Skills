# 리뷰 체크리스트

## 공식 문서 확인

다음 질문에 모두 답할 수 있을 때까지 필요한 공식 문서를 확인한다.

- 이번 요구사항에 새 기술, API, 프레임워크 제약이 있는가
- 최근 변경 가능성이 있는 정보인가
- 프로젝트가 이미 쓰는 기술의 공식 가이드 또는 1차 문서가 있는가
- 계획에 넣을 제약, 권장 패턴, 마이그레이션 포인트가 문서로 확인되었는가

공식 문서 확인 결과는 계획서의 "확인한 공식 문서와 기술 판단" 섹션으로 정리한다.

## 적절성 검토

초안 계획을 읽고 아래 finding이 있는지 점검한다.

- 요구사항의 목표 또는 성공 기준이 빠져 있다.
- 프로젝트 범위가 실제 파일, 모듈, 설정과 연결되지 않는다.
- 필요한 기술 검토가 빠졌거나 공식 문서 근거가 없다.
- 구현 단계가 모호해서 누가 봐도 바로 실행하기 어렵다.
- 검증 계획이 없거나 회귀 위험을 다루지 않는다.
- 리스크나 미확인 사항이 빠져 있다.
- `Unknowns`가 blocker 여부와 다음 확인 액션까지 구분되지 않는다.

finding이 하나라도 있으면 다음 순서로 수정한다.

1. finding을 짧고 구체적으로 적는다.
2. 계획서의 해당 섹션을 수정한다.
3. 같은 체크리스트를 다시 돌린다.

추가 finding이 없어질 때까지 반복한다.

## 과잉 설계 검토

적절성 검토가 끝난 뒤, 계획이 요구사항 대비 과하지 않은지 점검한다.

- 각 단계가 현재 요구사항과 직접 연결되는가
- "있으면 좋음" 수준의 리팩터링이 기본 계획에 섞여 있지 않은가
- 불필요한 추상화, 플랫폼 확장, 범용화가 들어가 있지 않은가
- 병렬 작업 스트림이 정말 필요한가
- 테스트/검증 범위가 요구사항 대비 과도하게 넓지 않은가
- 미래 가능성 대응이 현재 요구사항보다 앞서 있지 않은가

finding이 있으면 다음 순서로 수정한다.

1. 왜 과한지 한 문장으로 적는다.
2. 해당 항목을 축소, 삭제, out-of-scope 이동 중 하나로 처리한다.
3. 체크리스트를 다시 돌린다.

추가 finding이 없어질 때까지 반복한다.

## 종료 조건

아래 조건을 모두 만족하면 계획을 확정한다.

- 적절성 검토에서 추가 finding이 없다.
- 과잉 설계 검토에서 추가 finding이 없다.
- blocker unknown이 남아 있지 않다.
- 남아 있는 unknown은 blocker가 아니고 다음 확인 액션이 적혀 있다.
- 계획 단계마다 작업 범위와 검증 방법이 보인다.
- 최종 문서를 프로젝트 루트 `docs/`에 기록할 준비가 끝났다.
