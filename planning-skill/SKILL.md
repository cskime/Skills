요구사항을 바로 구현으로 옮기지 말고, 검증 가능한 구현 계획으로 정리한다. 계획이 확정되기 전에는 계획 작업에만 집중하고 코드 변경은 하지 않는다. 예외는 저장 경로인 프로젝트 루트 `docs/` 작성뿐이다.

## Workflow

1. 가능하면 즉시 Plan mode로 전환한다.
2. 런타임에서 명시적 mode 전환을 지원하지 않으면 `update_plan` 또는 동등한 계획 추적 수단으로 상태를 유지하고, 구현 대신 계획 작성에만 집중해 Plan mode처럼 동작한다.
3. 요구사항에서 목표, 제약, 성공 기준, 암묵적 가정을 추출한다.
4. 막히는 정보가 하나라도 있으면 가장 작은 질문만 한다. 질문 없이 진행할 수 있으면 보수적으로 가정하고 계획서에 명시한다.
5. 필요한 기술, 프레임워크, 라이브러리, 플랫폼 제약을 식별한다.
6. 공식 문서 또는 1차 자료만 사용해 필요한 기술 정보를 확인한다. 정보가 최근에 바뀔 수 있으면 반드시 최신 공식 문서를 확인한다.
7. 현재 프로젝트를 읽어 실제 변경 범위를 파악한다. 영향을 받는 디렉터리, 파일, 설정, 테스트, 데이터 모델, 외부 의존성을 구체적으로 적는다.
8. 초안 계획은 `references/plan-template.md` 구조를 따른다.
9. 초안 작성 후 `references/review-checklists.md`의 "적절성 검토"를 수행하고 finding이 남지 않을 때까지 수정한다.
10. 그다음 같은 파일의 "과잉 설계 검토"를 수행하고 finding이 남지 않을 때까지 수정한다.
11. blocker 성격의 unknown이 남아 있으면 계획을 최종본으로 확정하지 않는다. 가장 작은 확인 질문 또는 다음 조사 액션을 정리한 뒤 다시 검토한다.
12. blocker가 해소되고 검토가 끝난 계획만 최종본으로 취급한다.
13. 최종본을 프로젝트 루트 `docs/<date>-<topic>-plan.md` 형식으로 기록한다. `topic`은 요구사항의 핵심 주제를 짧은 slug로 정한다.
14. 프로젝트 루트 `docs/`가 없으면 생성한다.
15. Plan mode를 지원하는 런타임이면 문서 기록이 끝난 뒤에만 Plan mode를 해제한다.

## Execution Rules

- 사용자 언어를 따른다. 한국어 요구사항이면 한국어로 계획서를 작성한다.
- 공식 문서 확인 결과는 기술 선택 또는 제약과 연결해서 요약한다. 링크 나열만 하지 않는다.
- 작업 범위는 실제 프로젝트 구조에 근거해 적는다. 추측성 모듈 추가를 계획에 넣지 않는다.
- 구현 단계는 순서가 있고 검증 가능한 단위로 쪼갠다. 단계 수는 고정하지 않고 요구사항 크기에 맞춰 줄이거나 늘린다.
- 테스트, 마이그레이션, 릴리스 영향이 있으면 계획에 포함한다.
- 요구사항을 만족하는 최소 변경 경로를 기본안으로 삼는다.
- 현재 요구사항과 직접 관련 없는 리팩터링, 추상화, 플랫폼 확장은 out-of-scope로 분리한다.
- 계획이 충분히 명확하면 구현 예시 코드까지 쓰지 않는다. 이 스킬의 결과물은 계획 문서다.
- 같은 요구사항에 대한 기존 계획 문서가 있으면 우선 갱신 여부를 검토한다. 새 파일이 필요하면 기존 문서를 덮어쓰지 않도록 slug를 조정한다.
- `Unknowns`는 blocker 여부와 다음 확인 액션이 드러나게 적는다. blocker가 남아 있으면 최종본으로 닫지 않는다.
