---

요구사항을 바로 구현으로 옮기지 말고, 검증 가능한 구현 계획으로 수렴시킨다. 이 스킬은 planning 작업을 작은 `autoresearch` 루프로 운영한다.

## Operating Model

- 고정 하네스: 요구사항, 공식 문서/1차 자료, 실제 프로젝트 구조, 대상 사용자 또는 운영 주체, `references/plan-template.md`, `references/review-checklists.md`
- 변경 가능한 산출물: 현재 작업 중인 계획 문서 1개
- Keep 기준: blocker 또는 finding 감소, 실행 단계/검증 방식 명확화, 사용자 가치와 개발 의도 선명화, 같은 범위에서의 단순화
- Discard 기준: 근거 없는 추측 증가, scope inflation, blocker를 줄이지 못하는 복잡도 증가, 검증 계획 약화
- 종료 조건: 적절성 검토 0 finding, 과잉 설계 검토 0 finding, 독립 최종 재검토 0 finding, blocker 0

계획이 확정되기 전에는 계획 작업에만 집중하고 코드 변경은 하지 않는다. 예외는 저장 경로인 프로젝트 루트 `docs/` 작성뿐이다.

## Workflow

1. 요구사항을 받으면 계획이 확정될 때까지 구현 대신 계획 작성과 검증에만 집중한다.
2. 필요하면 `update_plan` 또는 동등한 계획 추적 수단으로 현재 상태를 유지한다.
3. 요구사항에서 목표, 제약, 성공 기준, 암묵적 가정, 변경하면 안 되는 고정 제약을 추출해 planning 하네스를 세운다.
4. 대상 사용자 또는 운영 주체, 현재 pain point, 왜 지금 이 변경이 필요한지, 구현 목적, 구현 후 기대하는 행동 변화 또는 관찰 가능한 결과를 먼저 정리한다. 직접적인 end-user UI가 없는 작업이면 개발자 또는 운영자 경험 관점으로 같은 질문에 답한다.
5. 사용자가 무엇을 더 쉽게, 빠르게, 안전하게 하게 될지와 이번 변경에서 절대 해치면 안 되는 경험적 guardrail을 적는다.
6. 막히는 정보가 하나라도 있으면 가장 작은 질문만 한다. 질문 없이 진행할 수 있으면 보수적으로 가정하고 계획서에 명시한다.
7. 필요한 기술, 프레임워크, 라이브러리, 플랫폼 제약을 식별한다.
8. 공식 문서 또는 1차 자료만 사용해 필요한 기술 정보를 확인한다. 정보가 최근에 바뀔 수 있으면 반드시 최신 공식 문서를 확인한다.
9. 현재 프로젝트를 읽어 실제 변경 범위를 파악한다. 영향을 받는 디렉터리, 파일, 설정, 테스트, 데이터 모델, 외부 의존성을 구체적으로 적는다.
10. 같은 요구사항에 대한 기존 계획 문서가 있으면 우선 그 문서를 읽고 갱신할지 판단한다. 기존 문서를 수정하면 그 문서를 현재 작업본으로 삼고, 새 파일이 필요하면 덮어쓰지 않도록 slug를 조정한다.
11. 새로 작성하는 계획이든 기존 계획 수정본이든 작업본은 `references/plan-template.md` 구조를 따른다. planning 단계에서 수정 가능한 산출물은 이 작업본 하나뿐이라고 가정한다.
12. 먼저 baseline 계획을 만든다. 문제 맥락, 개발 의도, 사용자 경험 목표, 고정 제약, 작업 범위, 구현 단계, 검증 계획, 리스크, unknowns를 빠짐없이 채운다.
13. 작업본에 대해 `references/review-checklists.md`의 "적절성 검토"를 수행하고 finding을 짧게 기록한다.
14. finding을 줄이거나 실행 가능성을 높이는 수정만 keep한다. blocker를 줄이지 못하고 문서만 길게 만드는 수정은 discard한다.
15. 이어서 같은 작업본에 대해 같은 파일의 "과잉 설계 검토"를 수행하고, 범위를 부풀리는 단계나 추상화는 축소 또는 제거한다.
16. 문서 구조, 구현 단계, 고정 제약, 검증 계획, `Unknowns`, `Blockers`, `Out of Scope` 중 하나라도 바뀌면 이전 clean 상태를 무효화하고 적절성 검토부터 다시 시작한다.
17. 적절성 검토와 과잉 설계 검토가 모두 clean이면, 이전 finding 메모를 그대로 답습하지 말고 요구사항, 공식 문서, 프로젝트 구조, 작업본을 처음 보는 것처럼 다시 읽는 "독립 최종 재검토"를 수행한다.
18. 독립 최종 재검토에서 새 finding이 하나라도 나오면 작업본을 수정한 뒤 적절성 검토부터 다시 반복한다. 첫 clean pass 하나만으로 종료하지 않는다.
19. 검토 중 문서를 수정했다면 종료 조건을 다시 확인하기 위해 적절성 검토, 과잉 설계 검토, 독립 최종 재검토를 필요한 만큼 반복한다. 새 계획 작성과 기존 계획 수정 모두 같은 최종 텍스트에서 추가 finding이 더 이상 나오지 않을 때까지 이 루프를 끝내지 않는다.
20. blocker 성격의 unknown이 남아 있으면 계획을 최종본으로 확정하지 않는다. 가장 작은 확인 질문 또는 다음 조사 액션을 정리한 뒤 다시 검토한다.
21. blocker가 해소되고 세 검토가 끝난 계획만 최종본으로 취급한다.
22. 최종본을 프로젝트 루트 `docs/<date>-<topic>-plan.md` 형식으로 기록한다. `topic`은 요구사항의 핵심 주제를 짧은 slug로 정한다.
23. 프로젝트 루트 `docs/`가 없으면 생성한다.
24. 문서 기록이 끝난 뒤에도 사용자에게 계획 경로, 핵심 결정, blocker 여부와 기대하는 사용자 경험 변화를 정리해 전달할 때까지는 계획 작업 범위를 유지한다.

## Execution Rules

- 사용자 언어를 따른다. 한국어 요구사항이면 한국어로 계획서를 작성한다.
- 계획서는 기술 변경 나열 전에 문제 맥락, 개발 의도, 구현 목적, 기대 목표를 먼저 고정한다.
- 직접 사용자 경험이 없는 작업이라도 영향을 받는 개발자, 운영자, 관리자 중 누가 primary user인지 정하고 그 관점에서 pain point와 기대 효과를 적는다.
- 공식 문서 확인 결과는 기술 선택 또는 제약과 연결해서 요약한다. 링크 나열만 하지 않는다.
- 작업 범위는 실제 프로젝트 구조에 근거해 적는다. 추측성 모듈 추가를 계획에 넣지 않는다.
- 각 섹션과 각 구현 단계는 요구사항, 공식 문서, 실제 프로젝트 구조 중 무엇에 근거하는지 드러나야 한다.
- 고정 하네스와 변경 가능한 산출물을 구분한다. planning 단계에서 수정 가능한 것은 계획 문서뿐이다.
- 구현 단계는 순서가 있고 검증 가능한 단위로 쪼갠다. 단계 수는 고정하지 않고 요구사항 크기에 맞춰 줄이거나 늘린다.
- 각 구현 단계는 어떤 요구사항, 성공 기준, 사용자 경험 목표를 닫는지 보여야 한다.
- 한 번에 한 종류의 finding을 줄이는 방향으로 문서를 수정한다. 수정 이유가 keep 기준에 맞지 않으면 되돌린다.
- 테스트, 마이그레이션, 릴리스 영향이 있으면 계획에 포함한다.
- 검증 계획에는 기능 동작 확인뿐 아니라 사용자 시나리오 기준의 수동 확인, empty/loading/error 상태, 접근성, 문구/피드백, 응답성 검토를 해당되는 범위에서 포함한다.
- 요구사항을 만족하는 최소 변경 경로를 기본안으로 삼되, 의도한 사용자 경험이나 운영 경험을 훼손하는 지름길은 최소안으로 취급하지 않는다.
- 현재 요구사항과 직접 관련 없는 리팩터링, 추상화, 플랫폼 확장은 out-of-scope로 분리한다.
- 계획이 충분히 명확하면 구현 예시 코드까지 쓰지 않는다. 이 스킬의 결과물은 계획 문서다.
- 새 계획 작성과 기존 계획 수정 모두 동일한 검토 기준을 적용한다. 문서를 한 번 수정할 때마다 필요한 체크리스트를 다시 돌리고, 추가 finding이 더 이상 나오지 않을 때만 다음 단계로 넘긴다.
- 첫 0-finding 패스를 종료 신호로 쓰지 않는다. 같은 최종 텍스트가 독립 최종 재검토까지 통과해야 한다.
- 독립 최종 재검토는 이전 finding 목록을 복기하는 방식이 아니라, 요구사항과 계획서를 처음 다시 매칭하듯 수행한다.
- 문서의 큰 구조나 단계, 검증, unknowns가 바뀌면 이전 clean 상태는 무효다. 적절성 검토부터 다시 시작한다.
- `적절히`, `필요시`, `관련 코드`, `후속 검토`처럼 실행 기준이 없는 모호한 표현은 finding으로 취급한다.
- `Unknowns`는 blocker 여부와 다음 확인 액션이 드러나게 적는다. blocker가 남아 있으면 최종본으로 닫지 않는다.
- `사용자가 편해진다`, `경험이 좋아진다`처럼 측정 불가능한 표현만 있고 어떤 행동 변화나 관찰 가능한 결과가 적혀 있지 않으면 finding으로 취급한다.
- finding이 0이어도 scope inflation이나 근거 없는 가정이 생겼다면 keep하지 않는다.
